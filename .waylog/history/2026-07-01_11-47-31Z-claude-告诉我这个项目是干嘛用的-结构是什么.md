## NVR AI Platform 项目介绍

这是一个**轻量级 NVR（网络视频录像机）智能分析平台**，核心用途是连接 RTSP 摄像头，实时预览视频流，并利用 AI 进行异常检测和内容检索。

### 主要功能

1. **实时 RTSP 拉流与预览** — 通过 OpenCV 或 ffmpeg 子进程解码摄像头流
2. **异常检测** — 检测花屏（HSV 分析）和绿线（霍夫直线检测）等画面异常
3. **AI 目标检测** — 使用 YOLOv8 实时检测视频中的人、车等目标，并带误报过滤
4. **文字搜图** — 基于 CLIP + FAISS 提取帧特征，支持用自然语言（如"穿红衣服的人"）回溯检索历史视频帧
5. **PyQt5 图形界面** — 预览窗口、搜索面板、告警面板一体化

### 目录结构

```

│   ├── feature_extractor.py # CLIP 模型提取帧特征
```

### 工作流程

```
RTSP 摄像头 → stream_capture → frame_extractor → yolo_detector（目标检测）
                                                     → green_line_detector（异常检测）
                                                     → feature_extractor（CLIP 特征）
                                                          → index_builder（FAISS 索引）
                                                               → searcher（文字检索）
                                        → video_widget（GUI 预览）
```

简单来说，这是一个**给安防摄像头加 AI 大脑**的项目 — 拉取视频流后，实时跑 YOLO 检测目标、检测画面异常，同时对视频帧建立 CLIP 特征索引，后续能用文字搜到想看的关键画面。

## 多线程重构方案

### 现状诊断

当前代码的问题：

| 问题 | 原因 |
|------|------|
| 搜索时界面卡顿 | `_do_search()` 用 `threading.Thread` 执行，但 CLIP 推理仍在主进程 GIL 内竞争 |
| 索引构建假死 | `_build_video_index()` 用 `threading.Thread`，无进度反馈，UI 无法感知 |
| 预览掉帧 | `_poll_frame()` 在主线程 QTimer 中，既要显示又要入队检测，堵塞 GUI 事件循环 |
| 退出不可靠 | 各线程用 `daemon=True` 简单粗暴，无顺序退出机制 |

### 目标架构

```
┌─────────────────────────────────────────────────────┐
│                   MainWindow (主线程)                  │
│  - 只负责：接收信号 → 更新 UI                          │
│  - QTimer 15fps 刷新 VideoWidget                      │
│  - 不执行任何耗时操作                                  │
└──────┬──────────────────┬──────────────────┬─────────┘
       │ frame_ready      │ search_*         │ progress_*
       ▼                  ▼                  ▼
┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│PreviewThread │  │SearchThread  │  │ IndexThread  │
│ (QThread)    │  │ (QThread)    │  │ (QThread)    │
│              │  │              │  │              │
│ RTSP 拉流     │  │ CLIP 编码     │  │ 视频读取      │
│ OpenCV 解码   │  │ 相似度计算    │  │ CLIP 编码     │
│ 坏帧过滤      │  │ torch.no_grad│  │ FAISS 构建    │
│ 30fps 拉流    │  │              │  │ 支持取消      │
└──────────────┘  └──────────────┘  └──────────────┘
```

### 三个线程详细设计

**PreviewThread**
- 封装原 `StreamCapture._run()` 逻辑
- 循环拉流 → 验证 → `.copy()` → emit `frame_ready(ndarray)`
- 支持 CPU/GPU 模式切换（通过 `set_decoder_mode()` slot）
- stop() → 设置停止标志 → 释放 cap → quit() → wait()

**SearchThread**（每次搜索新建）
- 接收：当前帧 + 文本 query + top_k
- 复用 `CLIPFeatureExtractor`（模型只加载一次）
- run：image encode → text encode → dot product → 排序
- emit `search_finished([{similarity, timestamp, frame}, ...])`

**IndexThread**（每次构建新建）
- 接收：video_path, sample_fps
- run：逐帧读取 → 采样 → CLIP 编码 → FAISS add → save
- 每 10 帧 emit `progress_updated(current, total)`
- 检查 `_cancelled` 标志以支持取消

### 优雅退出顺序

```
用户点击停止 / 关闭窗口
    ↓
1. PreviewThread.stop() → quit() → wait(5000)
2. SearchThread.stop()（如果正在搜索）→ quit() → wait(3000)
3. IndexThread.stop()（如果正在构建）→ quit() → wait(5000)
4. 主线程清理资源
```

## 调试重点

### 1. 线程生命周期

| 重点 | 文件 | 关键方法 |
|------|------|----------|
| PreviewThread 是否正常启动/停止 | [preview_thread.py](core/preview_thread.py) | `run()` / `stop()` / `_release_resources()` |
| SearchThread 每次搜索新建，搜索完自动回收 | [search_thread.py](core/search_thread.py) | `run()` / `stop()` |
| IndexThread 构建中途取消是否优雅退出 | [index_thread.py](core/index_thread.py) | `run()` / `cancel()` |

**验证点**：停掉预览后终端应打印 `线程已退出`，且没有残留进程。

### 2. 信号/Slot 通信

| 信号 | 发送方 | 接收方 | 传什么 |
|------|--------|--------|--------|
| `frame_ready` | PreviewThread | MainWindow | `np.ndarray`（深拷贝） |
| `search_finished` | SearchThread | MainWindow | `list[dict]` |
| `progress_updated` | IndexThread | MainWindow | `(int, int)` |

**验证点**：信号是否触发、是否在主线程执行 Slot、传过来的数据是否是拷贝而非引用。

### 3. 数据一致性

- `_on_search_finished()` 读字段 `"similarity"` 和 `"score"` 两兼容（刚才修的那个 bug）
- 帧传递用 `.copy()` 深拷贝，避免 PreviewThread 的帧被主线程修改导致 BGR→RGB 转换时崩溃
- `RealtimeIndexer` 在主线程持有，SearchThread（实时模式）只读取

### 4. 优雅退出顺序

先停显示 Timer → 停检测线程 → 停 PreviewThread → 取消 SearchThread → 取消 IndexThread → 释放资源。顺序错可能导致 `QThread: Destroyed while thread is still running` 警告。

### `.copy()` 做了什么

```python
frame_copy = frame.copy()
```

这一行在内存里**开辟了一块全新的区域**，把原始数据逐字节复制过去：

```
调用 copy() 前：
  frame       →  [内存地址 0x1000]  ← PreviewThread 持有引用
                  [89, 134, 200, 90, 135, 201, ...]  (640×480×3 ≈ 92万字节)

调用 copy() 后：
  frame       →  [内存地址 0x1000]  ← PreviewThread 持有
                  [89, 134, 200, 90, 135, 201, ...]
  
  frame_copy  →  [内存地址 0x7000]  ← 全新的内存块，通过信号传到主线程
                  [89, 134, 200, 90, 135, 201, ...]  ← 数据一模一样，但地址不同
```

### 如果不 copy 会怎样

```python
# ❌ 危险：两个变量指向同一块内存
frame_copy = frame          # 只是复制了"指针"，不是数据

# PreviewThread 写入下一帧 → 覆盖了 0x1000 的数据
# 主线程正在读 0x1000 → 读到一半旧一半新的撕裂画面
```

```
时间线：
  t0: PreviewThread 写入帧#1 到 0x1000 ─┐
  t1: 信号发出，主线程开始读 0x1000 ────┤ 同一个地址
  t2: PreviewThread 写入帧#2 到 0x1000 ─┘ 主线程读到一半被覆盖！
  t3: 主线程继续读 → 上半帧是帧#1，下半帧是帧#2 → 撕裂/崩溃
```

### 本质原因

`np.ndarray` 是**引用类型**。Python 里 `=` 赋值只复制引用（指针），不复制数据。

```python
a = np.array([1, 2, 3])
b = a               # b 和 a 指向同一块内存
b[0] = 999
print(a)            # [999, 2, 3]  ← a 也变了！

c = a.copy()        # c 是独立副本
c[0] = 0
print(a)            # [999, 2, 3]  ← a 不受影响
```

所以这句 `frame.copy()` 本质上是**用内存换安全**——每次多花 ~1MB 内存和几十微秒的拷贝时间，换来两个线程互不干扰，永远不会出现撕裂或悬空指针崩溃。

## 一、变量不是"盒子"，是"标签"

很多教程说"变量是装数据的盒子"，**这是错的**。实际上：

```python
a = 42
```

不是"把 42 放进盒子 a"，而是**在内存里创建了对象 42，然后给这个对象贴了个标签叫 a**。

```
内存：
  ┌──────────┐
  │   42     │  ← 对象
  └──────────┘
       ↑
       a    ← 标签（变量名指向这个对象）
```

用项目里的代码验证：

```python
import numpy as np

a = np.array([1, 2, 3])   # 创建一个数组对象
print(id(a))               # → 2814736403408  ← 内存地址
b = a                      # b 也指向同一个对象
print(id(b))               # → 2814736403408  ← 和 a 一样的地址！
```

**`id()` 告诉你对象在内存里的位置**，a 和 b 的 id 相同，说明它们不是两个数组，而是同一个数组挂了两个标签。

## 二、引用 vs 拷贝

```python
# ── 引用（共享同一块内存）─────────────────
a = np.array([1, 2, 3])

b = a               # b 仅仅是 a 的别名，指向同一个数组
b[0] = 999
print(a)            # [999, 2, 3]  ← a 也被改了！

# ── 浅拷贝（只复制第一层）─────────────────
c = a.view()        # 还是同一块数据
c = a[:]            # 同上

# ── 深拷贝（完全独立）─────────────────────
d = a.copy()        # 全新的数组，独立的内存
d[0] = 0
print(a)            # [999, 2, 3]  ← a 完全不受影响
```

**项目里的实际场景**（[preview_thread.py](core/preview_thread.py)）：

```python
# PreviewThread 里拉到的帧
frame = self.cap.read()      # frame 指向 OpenCV 内部的一块缓冲区

# 如果不 copy，直接发信号：
self.frame_ready.emit(frame)
# 主线程用这个 frame 时，PreviewThread 已经读下一帧覆盖了那块缓冲区
# → 画面撕裂 / 崩溃

# 所以必须 copy：
frame_copy = frame.copy()    # 开辟新内存，把像素数据全复制过去
self.frame_ready.emit(frame_copy)
# 主线程收到的是独立副本，PreviewThread 再怎么覆盖缓冲区都不影响
```

图解：

```
不 copy：                         copy 后：
 PreviewThread ↘                 PreviewThread → [自己的帧副本]
                  [同一块内存]                          ↕ 互不干扰
 主线程 ↗                        主线程       → [独立的帧副本]
```

## 三、进程和线程

### 进程（Process）

打开一个程序就是一个进程。**进程之间内存隔离**，A 进程访问不到 B 进程的变量。

```
┌─ 进程1: Google Chrome ─┐   ┌─ 进程2: VS Code ───┐
│  变量 a = 100           │   │  变量 a = "hello"   │
│  互不相干               │   │  互不相干            │
└─────────────────────────┘   └─────────────────────┘
```

### 线程（Thread）

**一个进程里可以有多个线程。线程之间共享同一块内存。**

```
┌──── 进程: NVR AI Platform ────────────────────┐
│                                                │
│  ┌─ 主线程 ─┐  ┌─ PreviewThread ─┐            │
│  │ 画 GUI   │  │ RTSP 拉流       │            │
│  │ 显示帧   │  │ 发 frame_ready  │            │
│  └──────────┘  └─────────────────┘            │
│       ↕ 共享 frame 数据 ↕                      │
│                                                │
│  共享的好处：传数据快，不用复制                  │
│  共享的危险：容易冲突（所以需要 .copy()）        │
└────────────────────────────────────────────────┘
```

项目里三种线程的使用方式：

```python
# 1. QThread —— 有信号/Slot，适合和 GUI 配合
class PreviewThread(QThread):
    frame_ready = pyqtSignal(np.ndarray)  # 线程间通信全靠这个

    def run(self):
        while not self._stopped:
            frame = self.cap.read()
            self.frame_ready.emit(frame.copy())  # 发给主线程

# 在主线程里启动
self._preview_thread = PreviewThread()
self._preview_thread.start()   # start() 会自动调用 run()


# 2. threading.Thread —— 普通后台线程，没有信号
self._process_thread = threading.Thread(
    target=self._process_loop,   # 要执行的函数
    daemon=True                  # daemon=True → 主程序退出时自动 kill
)
self._process_thread.start()    # 开始跑


# 3. QTimer —— 定时器，在主线程的事件循环里按固定间隔执行
self._display_timer = QTimer()
self._display_timer.timeout.connect(self._refresh_display)  # 每 1/15 秒执行一次
self._display_timer.start(1000 // 15)  # 66ms 间隔
```

## 四、线程间怎么通信

### 方式1：信号/Slot（QThread 专用，本项目用的就是这个）

```python
# PreviewThread 里：
self.frame_ready.emit(frame_copy)       # 发送
self.status_updated.emit("已连接")       # 发送

# MainWindow 里：
@pyqtSlot(np.ndarray)
def _on_frame_received(self, frame):    # 接收（在主线程执行）
    self._latest_frame = frame          # 存起来等 QTimer 显示
```

**信号发出后，PyQt 自动把数据跨线程传到主线程，Slot 在主线程执行。** 你不用管线程切换，框架帮你处理好了。

### 方式2：队列（daemon thread 用这个）

```python
# 主线程往队列放帧
self._process_queue.put_nowait((ts, frame))

# 后台线程从队列取帧
def _process_loop(self):
    while self._process_running:
        item = self._process_queue.get(timeout=0.5)  # 阻塞等，最多 0.5s
        ts, frame = item
        # 做 YOLO 检测...
```

队列就像一个**线程安全的管道**，一头放、另一头取，不会冲突。

### 为什么不直接用共享变量？

```python
# ❌ 错误的做法：共享变量 + 轮询
self._shared_frame = None    # PreviewThread 写, 主线程读

# PreviewThread：
while True:
    self._shared_frame = cap.read()     # 写

# 主线程 QTimer：
def _refresh(self):
    f = self._shared_frame              # 读
    # 问题：写和读可能同时发生，读到一半数据 → 崩溃
```

```
时间 →
  PreviewThread:  开始写帧#2 [████████████░░░░░░░░]
  主线程:            读取中...  [████████████░░░░░░░░]
                                    ↑ 这时候数据只写了一半！
```

## 五、内存是怎么分配的

```python
import numpy as np

frame = np.zeros((480, 640, 3), dtype=np.uint8)

# 内存布局：
# 480 × 640 × 3 × 1字节 = 921,600 字节 ≈ 900 KB
# 这些字节在内存里是连续排列的（C-order）：
#
# [B,G,R, B,G,R, B,G,R, ...]  ← 第0行
# [B,G,R, B,G,R, B,G,R, ...]  ← 第1行
# ...
# [B,G,R, B,G,R, B,G,R, ...]  ← 第479行
#
# frame[10, 20, 0]  → 访问内存偏移 = (10×640 + 20)×3 + 0
#  行=10, 列=20, 通道0(B)
```

`uint8` = 无符号8位整数，范围 0~255，正好是像素值的范围。

## 1. `[hevc] Skipping invalid undecodable NALU: 1`

```
NALU = 视频压缩后的一小段数据包
```

HEVC（H.265）编码把一帧画面拆成很多个 NALU（网络抽象层单元）。RTSP 通过网络传输时，部分数据包丢了/坏了，解码器收到后说"这段数据不完整，跳过"。

**类比**：你在下载一个 100 页的 PDF，第 35 页只下了一半，软件说"这页损坏了，跳过"。不影响其他能正常显示的页。

## 2. `Could not find ref with POC 35`

```
POC = Picture Order Count = 帧的"编号"
ref = 参考帧
```

HEVC 压缩视频不会每帧都存完整画面，而是**只存"这帧和上一帧哪里不同"**。如果"上一帧"（参考帧）因为网络丢包丢失了，解码器就没法重建当前帧。

```
时间轴上的帧：
  帧#34  帧#35  帧#36  帧#37
  ✓正常  ✗丢了  ?需要帧#35当参考  ?需要帧#35当参考
                    ↑ 找不到帧#35 → 报错
```

**这是网络问题，不是代码 bug**。RTSP 走 UDP 天然会丢包，不影响后续帧的解码。

## 3. `First slice in a frame missing`

一帧画面可能被切成多个 **slice**（切片）分别压缩传输。第一个 slice 丢了，后面的也就没法拼起来。解码器说"第一片都没到，整帧跳过"。

## 4. `0: 192x640 (no detections)`

```
0:         → 第 0 张图
192x640   → YOLO 把输入缩放到 192×640 像素做推理
(no detections) → 这一帧里没检测到目标（人、车等）
```

这是 YOLO 的正常输出，每帧跑一次，告诉你"画面上有东西吗？——没有"。

## 5. `Speed: 2.6ms preprocess, 53.5ms inference, 2.0ms postprocess`

```
preprocess  = 2.6ms   → 把帧缩放、归一化，准备喂给模型
inference   = 53.5ms  → YOLO 推理（最耗时的部分）
postprocess = 2.0ms   → 解析推理结果，画出检测框
────────────────────
一帧总共 ≈ 58ms，也就是约 17 帧/秒的处理速度
```

**断点在于**：daemon 线程算出了 `ok_dets`（检测到的目标列表），但**没有路径把它传回主线程显示**。

## 修复方案

加一条数据回流：

```
daemon 线程                        MainWindow._refresh_display()
     │                                    │
     ├─ self._latest_detections           │
     │  = ok_dets  ──────────→            ├─ 读 _latest_detections
     │  (简单赋值，GIL 保证原子性)          ├─ 在 frame copy 上画框
     │                                    ├─ video.set_frame(frame_with_boxes)
     │                                    └─ queue.put(frame) → daemon
```

**为什么不用锁也不用信号？** `list` 的简单赋值 `self._latest_detections = ok_dets` 在 Python 的 GIL（全局解释器锁）下是**原子操作**——同一时刻只有一个线程在执行 Python 字节码，不会有"写一半被读"的问题。

```
│  │   5. queue.put(frame)                   → 丢给 daemon 线程检测    │  │
│  └────────────────────────────────────────────────────────────────┘  │
│                                                                        │
│  _on_search()      → 新建 SearchThread → start()                      │
│  _on_index_video() → 新建 IndexThread  → start()                      │
│                                                                        │
│  检测器（在主线程初始化，daemon 线程使用）：                             │
│    self.detector  = YoloDetector()                                    │
│    self.fp_filter = FalsePositiveFilter()                              │
│    self.indexer   = RealtimeIndexer()  ← daemon 线程写入帧特征         │
└──────────────────────┬─────────────────────────────────────────────────┘
                       │ queue.Queue (线程安全管道)
                       ▼
┌─ daemon 线程 (threading.Thread) ───────────────────────────────────────┐
│  _process_loop():                                                      │
│    while _process_running:                                             │
│      ts, frame = queue.get(0.5s)                                       │
│      _process_frame(ts, frame):                                        │
│        1. green_line_detector.detect(frame)   → 绿线/花屏告警          │
│        2. yolo_detector.detect(frame, verbose=False) → 检测结果        │
│        3. fp_filter.filter(dets)               → 过滤误报              │
│        4. self._latest_detections = ok_dets   → 写入共享变量           │
│        5. realtime_indexer.add_frame(frame)   → CLIP 实时索引          │
└────────────────────────────────────────────────────────────────────────┘
```

### 三个共享数据的传递方式

| 数据 | 方向 | 方式 | 为什么安全 |
|------|------|------|-----------|
| 帧图像(frame) | PreviewThread→主线程 | `pyqtSignal(np.ndarray)` | PyQt 框架内部序列化 |
| 帧图像(frame) | 主线程→daemon | `queue.Queue` | 线程安全管道，无竞态 |
| 检测结果(list) | daemon→主线程 | `self._latest_detections = ok_dets` | GIL 保证赋值原子性 |

## 修复：画框不显示

**根本原因**：`_log()` 在 daemon 线程里操作 GUI 控件（`QTextEdit.append`），可能抛异常。之前的代码：
```python
if ok_dets:
    self._log(...)            # ← 这里抛异常了
self._latest_detections = ok_dets   # ← 这行永远没执行
```

**修复**：把赋值挪到 `_log` 前面，检测结果先存，日志后写。
